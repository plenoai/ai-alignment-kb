# 並列専門エージェントパターン

## 意図

単一の汎用エージェントではなく、ドメインに特化した複数のエージェントを並列実行することで、監査の速度・深度・相互検証性を同時に高める。

## 動機

汎用エージェントは広範なタスクをこなせる反面、専門領域での判断精度が落ちる。また直列処理では監査対象が増えるほど時間が線形に増加する。

人間の専門家チームが「分業して同時並行でレビューする」構造を AI で再現することで、スケールと品質を両立する。

## 構造

```
[オーケストレーター]
        │
        ├── [専門エージェント A] ── ドメイン A を並列分析
        ├── [専門エージェント B] ── ドメイン B を並列分析
        ├── [専門エージェント C] ── ドメイン C を並列分析
        └── ...
                │
        [結果の集約・照合]
                │
        [統合レポート]
```

### 専門化の軸

専門化の軸はコンテキストによって異なる：

| 軸 | 例 |
|----|----|
| プラットフォーム別 | Google Ads 専門, Meta 専門, LinkedIn 専門 |
| 機能別 | セキュリティ監査, パフォーマンス監査, コンプライアンス監査 |
| 時系列別 | 過去トレンド分析, 現状評価, 将来予測 |
| レイヤー別 | データ層, ロジック層, UI 層 |

## 帰結

**得られるもの**
- 処理時間の短縮（直列 → 並列）
- 各エージェントの専門性向上（スコープの狭さが精度をあげる）
- 相互検証：A エージェントの死角を B エージェントが補完できる
- 障害の局所化：1 エージェントの失敗が全体を止めない

**失うもの**
- オーケストレーションの複雑性が増す
- エージェント間の結果が矛盾した場合の調停ロジックが必要
- 「専門化」の設計を誤ると重複や空白が生じる

## 既知の使用例

- **Claude Ads:** 6 つの並列エージェントが 5 プラットフォームを同時分析（190 チェック）
- **コードレビュー:** セキュリティ, パフォーマンス, スタイルを別エージェントが並列チェック
- **論文査読支援:** 方法論・統計・倫理を専門家エージェントが並列評価
