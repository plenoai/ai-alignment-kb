# 定量ヘルススコアパターン

## 意図

AI の評価結果を単一の定量スコアに集約し、人間が素早く俯瞰・比較できる透明な指標を提供する。

## 動機

AI が生成する詳細レポートは情報量が多すぎて、人間が監視・意思決定に使いにくい。「良いのか悪いのか」が一目でわからない出力は承認ゲートとして機能しない。

定量スコアは情報の圧縮であり、人間の注意を「異常値」に向けるシグナルとして機能する。

## 構造

```
[チェック項目の評価結果]
         │
[重み付き集計]
  ├── Critical 項目（×3）
  ├── Major 項目（×2）
  └── Minor 項目（×1）
         │
[0-100 スコア]
         │
[グレード変換]  A(90+) / B(80-89) / C(70-79) / D(60-69) / F(<60)
         │
[アクション推奨]  緊急対応 / 改善推奨 / 維持
```

### スコア設計の原則

1. **単一数値 + グレード:** 数値の精度と直感的なグレードを両立する
2. **内訳の提示:** スコアの根拠を説明可能にする（説明可能性）
3. **アクション連動:** スコア帯ごとに推奨アクションを定義する
4. **ベースライン比較:** 絶対値より「前回比・業界平均比」が意味を持つことが多い

## 帰結

**得られるもの**
- 人間の認知負荷の低減（詳細を読まなくても全体把握が可能）
- 時系列比較・複数対象の比較が容易
- 承認ゲートの判断基準として機能する（スコア < X なら自動ブロック等）
- 優先順位付けが容易（スコアの低い順に対応）

**失うもの**
- 情報の圧縮による文脈の喪失（スコアだけ見て詳細を読まないリスク）
- 重み付けの設計が恣意的になりやすい
- スコアの最適化が目的化する（スコアゲーミング）
- 「A 評価だから安全」という過信を生む可能性

## 既知の使用例

- **Claude Ads:** Ads Health Score (0-100) + A〜F グレード、5 プラットフォームを一覧比較
- **コードカバレッジ:** テストカバレッジ % でコード品質を近似
- **PageSpeed Insights:** 0-100 スコアで Web パフォーマンスを可視化
- **信用スコア:** 複数の財務指標を統合した単一スコア
