# 外部化知識（RAG）

AI の判断に使う知識をモデルのパラメータではなく外部ファイルに置き、モデルの再学習なしに知識を更新・拡張できるようにする。

## コンテキスト

頻繁に変化するドメイン知識（業界規制、ベンチマーク数値、プラットフォーム仕様）を参照して評価を行う AI エージェントを構築している。知識の更新サイクルがモデルの学習サイクルより速い。

## 問題

**ファインチューニングはコストが高く更新頻度の高い知識に向かない。かといって知識をプロンプトにベタ書きすると管理・更新が破綻する。**

## フォース

- モデルへの知識の焼き込みはコストが高く、更新のたびに再学習が必要
- プロンプトへのベタ書きは管理しづらく、変更履歴が追えない
- 知識が多すぎるとコンテキストウィンドウを圧迫する
- 知識の品質管理責任を誰が持つかが不明確になりやすい
- 焼き込まれた知識は「なぜその知識があるか」が不透明になる

## 解決策

**知識をファイルとして外部管理し、エージェントが実行時にオンデマンドで読み込む。ファイルは git で管理することで、バージョン管理・差分確認・ロールバックを通常のソフトウェア開発と同じプロセスで行う。**

```
[外部知識ベース]
  ├── rules/
  ├── templates/
  └── reference-data/
         │
    オンデマンド読み込み
         │
    [AI エージェント]
         │
    推論・評価
```

### ファイルの種類

| 種類 | 内容 | 更新頻度 |
|------|------|---------|
| ルールファイル | チェック項目・判断基準 | 月次〜四半期 |
| テンプレート | 業種別・ユースケース別の設定ひな形 | 四半期〜年次 |
| 参照データ | ベンチマーク数値、業界標準値 | 週次〜月次 |

## 結果コンテキスト

**得られるもの**
- 知識の更新コストが低い（ファイル編集のみ）
- 知識の来歴が追跡可能（git 履歴）
- コンテキストに応じてロードする知識を選べる（効率化）
- 知識とモデルの関心分離

**失うもの**
- 毎回の読み込みによるレイテンシ増加
- ファイル管理の複雑性（命名規則、バージョニング、依存関係）
- 知識ベースの品質管理責任が人間に残る

## 既知の使用例

- **Claude Ads:** プラットフォーム別参照データをオンデマンドで読み込む RAG アーキテクチャ
- **Claude Code スキルシステム:** `.claude/skills/` に置いたマークダウンを都度読み込む
- **法律 AI:** 最新の法改正をベクター DB に格納し、クエリ時に参照
