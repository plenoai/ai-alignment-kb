# ドメイン知識コード化

人間の専門家が持つ暗黙知・経験則を、明示的なルールセット・チェックリストとしてコード化し、AI が一貫して適用できるようにする。

## コンテキスト

特定ドメインの深い知識（業界慣行、落とし穴、ベストプラクティス）を要する評価タスクを、AI エージェントに委ねようとしている。専門家は存在するが、評価の頻度・量が専門家のキャパシティを超えている。

## 問題

**AI モデル単体では、専門家が当然知っている暗黙の判断基準を持たない。専門家に毎回確認するとスケールしない。**

## フォース

- 専門家の判断基準は頭の中にあり、言語化されていないことが多い
- AI への曖昧な指示は、評価結果の一貫性を損なう
- 専門家をループに入れ続けるとスケールの利点が消える
- かといってルール化しすぎると、文脈依存の判断を取りこぼす
- ルールは陳腐化する——メンテナンスの責任者が必要になる

## 解決策

**「専門家の頭の中にある判断基準」をインタビューや事例収集で抽出し、優先度・重み付き のルールセット／チェックリストとして外部化する。AI エージェントはこのリストを参照して評価を行う。**

```
[専門家インタビュー / 事例収集]
         │
[ルール・チェック項目の抽出]
         │
[優先度・重み付けの定義]
         │
[チェックリスト / ルールセット]
         │
    [AI エージェント]  ←── このリストを参照して評価
```

### コード化の粒度

| 粒度 | 例 | 適用場面 |
|------|----|---------|
| バイナリチェック | 「コンバージョンタグが重複していないか」 | 明確な正誤がある場合 |
| スコアリング | 「品質スコア 7 以上のキーワードが 80% 以上か」 | グラデーションがある場合 |
| パターンマッチ | 「入札戦略が目標と整合しているか」 | 文脈依存の判断 |

## 結果コンテキスト

**得られるもの**
- 専門知識の民主化（専門家なしでも専門家水準の監査が可能）
- 一貫性（人間の判断バイアスや見落としを排除）
- 更新可能性（ルールセットを書き換えるだけで全エージェントに反映）
- 透明性（なぜそのスコアになったかが説明可能）

**失うもの**
- 「ルール化できない」暗黙知は取りこぼす
- ルールの陳腐化リスク（定期的なメンテナンスが必要）
- 新規ドメインでは初期コスト（知識収集・構造化）が高い
- 過度なルール化が「チェックボックス思考」を生む危険性

## 既知の使用例

- **Claude Ads:** 190 のチェック項目（Google Ads 74, Meta 46 等）を業種別にカスタマイズ
- **セキュリティスキャナー:** OWASP Top 10 をルールとして実装した静的解析ツール
- **医療診断支援:** 臨床ガイドラインをルールエンジンとして実装
