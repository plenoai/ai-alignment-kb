# Read-Write 境界

AI エージェントの「読み取り（観察）」と「書き込み（実行）」を明示的に分離し、不可逆な副作用を人間の確認ポイントで制御する。

## コンテキスト

AI エージェントが分析だけでなく実行（外部 API へのコミット、データの変更、リソースの削除）まで担いうる環境で、自律性と安全性を両立しようとしている。

## 問題

**AI による自律実行は速度と網羅性をもたらすが、誤った判断が不可逆な結果につながるリスクがある。全操作に人間確認を挟むとスケールしない。**

## フォース

- 全操作を自動化すれば速いが、誤実行の影響が大きい
- 全操作に人間確認を挟むと、自動化の利点が消える
- 操作によって「元に戻せる度」が大きく異なる
- 人間の確認疲れが起きると、形骸的な承認が増える
- 承認ゲートが多すぎると、AI のメリットより人間のボトルネックが目立つ

## 解決策

**操作の可逆性を基準に自律性の境界を引く。読み取り（観察・分析・レポート生成）は完全自動化し、書き込み（実行・変更・削除）は人間の承認ゲートを経由してから実行する。**

```
[AI エージェント]
     │
     ├─── READ（観察・分析）─── 完全自動化
     │         ↓
     │    調査レポート / 推奨事項
     │         ↓
     └─── WRITE（実行・変更）─── 人間承認ゲート ─── 実行
```

### 境界の判断基準

| 操作 | 可逆性 | 自律性 |
|------|--------|--------|
| データ取得・分析 | 高 | 完全自動 |
| ドラフト生成 | 高 | 完全自動 |
| 設定変更（ロールバック可能） | 中 | 承認後自動 |
| 外部 API へのコミット | 低 | 人間確認必須 |
| 予算変更・削除 | 低〜なし | 人間確認必須 |

## 結果コンテキスト

**得られるもの**
- 不可逆な失敗のリスクを局所化できる
- 読み取りフェーズでの自動化メリット（速度・網羅性）を享受できる
- 人間の認知負荷を「判断」に集中させられる

**失うもの**
- 完全自動化と比べてエンドツーエンドの速度が落ちる
- 承認ゲートの設計・運用コストがかかる
- 境界の引き方を誤ると形骸化する（全部「読み取り」扱いにするなど）

## 既知の使用例

- **Claude Code + Google Ads 監査:** 監査レポート生成は自動、ネガティブキーワード登録は人間確認後に API 実行
- **コードレビュー AI:** Lint・静的解析は自動、コードの自動マージは人間承認後
- **医療 AI 診断支援:** 症状分析・鑑別診断リスト生成は自動、処方は医師の最終判断
- **Ralph Loop:** リファクタリング後に E2E テストが失敗したら変更を自動リバート。書き込みをラウンド単位で可逆にする
- **Ankit Jain「コードレビューは死んだ」:** "ship fast, observe everything, revert faster" という新指標を提唱。可逆性を前提にした高速フィードバックループへの移行を主張。 → [Latent.space 事例](./latent-space-reviews-dead-case.md)
- **OpenClaw 自己強化エージェント:** `ops/coach` エージェントに "Never apply config changes automatically. Propose only" を非交渉ルールとして設定。`activeHours`（09:00–23:00）で稼働時間を制限し、人間がレビューできない時間帯には書き込みゲートが機能しない条件を排除する。 → [OpenClaw 事例](./openclaw-self-reinforcing-case.md)
- **Cursor クラウドエージェント:** エージェントは VM 上で自由にコードを書き込むが、成果物は diff ではなくログ・動画記録・ライブプレビューとして出力。人間は成果物ベースでレビューする。承認ゲートの入力がリッチ化した形態。 → [Cursor 事例](./cursor-third-era-case.md)
- **Symphony（OpenAI）:** エージェントはワークスペース内で自律的にコードを書くが、本番統合は PR マージという承認ゲートを経由。承認ポリシーを実装定義にすることで境界の位置を環境の信頼度に応じて調整可能。 → [Symphony 事例](./symphony-case.md)
