# 並列専門エージェント

単一の汎用エージェントではなく、ドメインに特化した複数のエージェントを並列実行することで、監査の速度・深度・相互検証性を同時に高める。

## コンテキスト

複数の専門領域にまたがる評価タスク（マルチプラットフォーム監査、マルチレイヤーレビュー等）を AI に委ねようとしている。単一エージェントで処理するか、複数エージェントに分割するかを決める段階にある。

## 問題

**汎用エージェントはスコープが広いほど判断精度が落ち、直列処理では監査対象が増えるほど時間が線形に増加する。**

## フォース

- 汎用エージェントはどの領域も「それなり」にこなすが、専門深度が浅い
- 専門化するほど精度は上がるが、統合・調整のコストが増す
- 直列処理は単純だが、対象規模に比例して時間がかかる
- 並列化すると結果の矛盾が生じたとき、どちらを信じるかが問題になる
- エージェントの専門化の軸を誤ると、重複や盲点が生まれる

## 解決策

**オーケストレーターがタスクを専門領域に分割し、各専門エージェントを並列実行する。各エージェントは自身の専門スコープに集中し、結果をオーケストレーターが集約・照合して統合レポートを生成する。**

```
[オーケストレーター]
        │
        ├── [専門エージェント A] ── ドメイン A を並列分析
        ├── [専門エージェント B] ── ドメイン B を並列分析
        ├── [専門エージェント C] ── ドメイン C を並列分析
        └── ...
                │
        [結果の集約・照合]
                │
        [統合レポート]
```

### 専門化の軸

専門化の軸はコンテキストによって異なる：

| 軸 | 例 |
|----|----|
| プラットフォーム別 | Google Ads 専門, Meta 専門, LinkedIn 専門 |
| 機能別 | セキュリティ監査, パフォーマンス監査, コンプライアンス監査 |
| 時系列別 | 過去トレンド分析, 現状評価, 将来予測 |
| レイヤー別 | データ層, ロジック層, UI 層 |

## 結果コンテキスト

**得られるもの**
- 処理時間の短縮（直列 → 並列）
- 各エージェントの専門性向上（スコープの狭さが精度をあげる）
- 相互検証：A エージェントの死角を B エージェントが補完できる
- 障害の局所化：1 エージェントの失敗が全体を止めない

**失うもの**
- オーケストレーションの複雑性が増す
- エージェント間の結果が矛盾した場合の調停ロジックが必要
- 「専門化」の設計を誤ると重複や空白が生じる

## 既知の使用例

- **Claude Ads:** 6 つの並列エージェントが 5 プラットフォームを同時分析（190 チェック）
- **コードレビュー:** セキュリティ, パフォーマンス, スタイルを別エージェントが並列チェック
- **論文査読支援:** 方法論・統計・倫理を専門家エージェントが並列評価
- **ゲームAI / キャラクターAI群:** メタAIの下で多数のNPCエージェントが並列に自律動作。メタAIがオーケストレーターとして全体を統括
