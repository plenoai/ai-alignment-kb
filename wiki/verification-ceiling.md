# 検証天井

自動検証基盤（テスト・CI・ヘルスチェック）の信頼性が、エージェントに許容できる自律性レベルの上限を決定する。

## コンテキスト

自律エージェントが「実行→検証→続行 or リバート」のループで動作している。エージェントの自律性を高めるほど、人間の代わりに自動検証が正しさの判断を担う割合が増える。

## 問題

**自律エージェントの能力を高めても、その実行結果を検証する自動テスト・CI が不安定だと、エージェントは正しい判断ができない。検証基盤の信頼性が、自律性のスケールを律速する。**

## フォース

- 自律性を上げるほど、人間のレビュー頻度が下がり、自動検証への依存度が上がる
- 自動検証が不安定（flaky）だと、エージェントは「成功」と「偶然の成功」を区別できない
- 並列実行するエージェント数が増えるほど、1 つの flaky テストの影響が乗算的に拡大する
- テスト基盤の改善は地味な作業であり、エージェント機能の開発より優先されにくい
- 検証基盤の不備は「エージェントが間違える」ではなく「全エージェントが同時に間違える」として顕在化する

## 解決策

**自律性レベルを上げる前に、そのレベルが前提とする自動検証の信頼性を確保する。検証基盤の信頼性がエージェントの自律性の天井であることを設計原則として明示する。**

```
検証信頼性      自律性レベル
───────────    ──────────
不安定な検証  →  L1–L2 が上限（人間が毎回確認）
安定した検証  →  L3 が可能（監視のみ）
高信頼な検証  →  L4 が可能（例外時のみ介入）
完全な検証    →  L5 が可能（完全自律）← 理論上の極限
```

### 天井を引き上げるための施策

| 施策 | 効果 | コスト |
|------|------|--------|
| flaky テストの根絶 | 検証結果を信頼可能にする | テスト基盤の継続的保守 |
| 環境の冪等性確保 | エージェント間の干渉を排除 | VM/コンテナの隔離投資 |
| 検証自体のメタ検証 | 検証の偽陽性・偽陰性を検知 | 二重検証のコスト |
| 段階的ゲート | 低信頼な検証段階で止め、高信頼な検証は後段で | パイプライン設計の複雑化 |

### 並列実行と天井の関係

単一エージェントでは許容できる flaky 率が、並列実行では許容不能になる：

```
flaky 率 1% × 1 エージェント → 1% の偽判定
flaky 率 1% × 50 エージェント → ほぼ毎回どこかで偽判定が発生
```

これが Cursor の「flaky テストのカスケード障害」の構造的原因。

## 結果コンテキスト

**得られるもの**
- 自律性の引き上げが「検証基盤の信頼性」という測定可能な前提条件に紐づく
- 「エージェントが使えない」問題の根本原因（検証基盤）を特定できる
- 検証基盤への投資の優先度を正当化できる（自律性の天井を直接引き上げる）

**失うもの**
- 検証基盤が不十分なうちは自律性を上げられない（速度の制約）
- 完全な検証は理論上達成不能であり、天井を完全に撤去することはできない
- 検証基盤の信頼性測定自体が難しい（「テストは十分に安定しているか？」の判断基準）

## 既知の使用例

- **Cursor クラウドエージェント:** flaky テストや壊れた環境が全エージェントにカスケード障害を引き起こすと報告。検証基盤の信頼性が自律クラウドエージェント（Era 3）の前提条件であることを実体験から認めている。 → [Cursor 事例](./cursor-third-era-case.md)
- **Spotify Honk（2025-12）:** バックグラウンドコーディングエージェント Honk は LLM-as-judge 検証レイヤーを持ち、全セッションの約 25% を拒否。拒否後にエージェントが自己修正できた割合は 50%。検証レイヤーがなければ不良コードの 1/4 がそのまま通過する。検証がゲートとして機能している実測値。 — [Spotify Engineering Blog](https://engineering.atspotify.com/2025/12/feedback-loops-background-coding-agents-part-3)
- **マルチエージェント本番障害（Augment Code 調査）:** マルチエージェントシステムの本番障害率は 41–86.7%。障害原因の 21.3% が検証ギャップ。PwC 事例ではバリデーション層の追加で精度が 10% → 70% に改善（7 倍）、ハルシネーション 40% 削減。 — [Augment Code](https://www.augmentcode.com/guides/why-multi-agent-llm-systems-fail-and-how-to-fix-them)
- **OpenAI Codex チーム — Harness Engineering（2026-02）:** 3 人のエンジニアが Codex 経由で約 1,500 PR をマージし、5 ヶ月で 100 万行超のコードを構築。構造テスト・カスタム linter で依存関係を機械的に強制。ただし Martin Fowler が「機能・振る舞いの検証が欠如している」と指摘 — 天井の存在を示す反面教師的事例。 — [OpenAI Blog](https://openai.com/index/harness-engineering/) / [Martin Fowler](https://martinfowler.com/articles/exploring-gen-ai/harness-engineering.html)
- **CodeScene（2025-26）:** 完全エージェント化後にタスク速度 2–3 倍を達成。前提としてユニットテストカバレッジ 99%、Code Health スコア 9.5/10 以上を維持。高信頼な検証基盤が速度向上の前提条件として機能。 — [CodeScene Blog](https://codescene.com/blog/agentic-ai-coding-best-practice-patterns-for-speed-with-quality)
- **Crawshaw sketch.dev（2025-26）:** エージェントによるコード生成率が 25%（2025-02）→ 90%（2026-02）に上昇。同時に開発者の時間配分がコード書き 50% → 5%、コード読み（検証） 50% → 95% に変化。自律性の上昇が人間の役割を「書く」から「検証する」に移行させた実測。 — [crawshaw.io](https://crawshaw.io/blog/eight-more-months-of-agents)
- **Ralph Loop:** E2E テストが安全網として機能するが、テスト自体が不安定だとリバート判断が誤る。テスト信頼性がクローズドループの自律改善品質の天井になっている。 → [Ralph Loop 事例](./ralph-loop-case.md)
