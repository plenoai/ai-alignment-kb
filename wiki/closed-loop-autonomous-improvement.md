# クローズドループ自律改善

AI エージェントが終了条件と自動安全網を持ちながら、改善サイクルを自律的に反復し、人間の介入なしにコードやシステムの品質を継続的に向上させる。

## コンテキスト

AI エージェントが問題発見・修正・品質改善を自律的に実行できる環境で、エージェントの暴走リスクを抑えつつ、人間の監視コストを最小化して継続的な品質向上を実現しようとしている。

## 問題

**AI が自律的に改善を繰り返す仕組みは強力だが、「いつ止まるか」「誤修正で品質が下がったとき誰が気づくか」が解決されなければ、人間は安心して実行を委ねられない。**

## フォース

- 人間が監視しながら回すと自動化のメリットが消える
- 無制限に動かすと誤った方向への修正が積み重なる
- 全ての修正に人間確認を挟むとスケールしない
- 何ラウンド回したかの記録がないと進捗が見えない
- テストが壊れた修正を見逃すとシステムが劣化する

## 解決策

**改善サイクルを「専門フェーズの連鎖」として定義し、終了条件と自動安全網（テスト＋自動リバート）を組み込んで自律実行する。人間は起動時にパラメータを渡すだけでよい。**

```
[人間: improve <time|rounds> を実行]
         │
         ▼
┌────────────────────────────────┐
│  Round N                       │
│                                │
│  QA ──→ Fix ──→ Refactor       │
│  (バグ発見   (Issue修正   (品質改善)
│   Issue起票)  Issue close)      │
│                                │
│  E2E テスト ──── 失敗? ──→ 自動 Revert
└────────────────────────────────┘
         │
    終了条件の評価
    ├── 指定時間を超過?
    ├── 指定ラウンド数に達した?
    └── Open Issue がゼロ? (早期終了)
         │
[人間: 翌朝、改善されたコードを確認]
```

### 終了条件の設計

| 条件 | 用途 |
|------|------|
| 時間上限（例: 120分） | 夜間バッチとして時間を区切る |
| ラウンド上限（例: 5ラウンド） | 明示的に回数を制御する |
| 品質ゲート（Open Issue = 0） | 問題解消を確認して早期終了 |

### 自動安全網の多重化

1. **E2E テスト:** 各ラウンド終了後に必ず実行
2. **自動リバート:** テスト失敗時は当該ラウンドの変更を自動で元に戻す
3. **振り返り記録:** 各ラウンドの結果を記録し、次ラウンドの改善手法に反映

## 結果コンテキスト

**得られるもの**
- 人間が寝ている間にコード品質が継続的に向上する
- 暴走リスクが自動リバートで局所化される（ラウンド単位の影響範囲）
- 終了条件により「どこまでやったか」が明確になる
- 人間の介入は起動時のパラメータ設定のみに集約される

**失うもの**
- 各修正の意図を人間がリアルタイムに把握できない
- E2E テストが不完全な場合、誤修正を検出できない
- 長時間の自律実行中は方向性の軌道修正が難しい
- ラウンド数・時間の設定を誤ると修正量が不適切になる

## 既知の使用例

- **Ralph Loop（Claude Code スキル）:** QA → Fix → Refactor の3フェーズを `improve 120`（2時間）や `improve --rounds 5`（5ラウンド）で自律実行。E2E テスト失敗時は自動リバート。 → [Ralph Loop 事例](./ralph-loop-case.md)
