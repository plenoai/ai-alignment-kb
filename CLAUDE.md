# Scalable Oversight Framework — 編集方針

## リポジトリの目的

AI エージェントに対する人間の監視（Oversight）を、品質を落とさずスケールさせるパターン集。分野横断的に事例を収集し、再利用可能なパターンとして抽象化する。

## ディレクトリ構成と役割

| ディレクトリ | 役割 | 追加基準 |
|------------|------|---------|
| `app/patterns/` | 再利用可能な設計パターン | 複数の事例で観察された抽象的な型 |
| `packages/wiki/` | 概念・用語の解説 | パターンを理解するための前提知識 |
| `packages/knowledge/` | 事例・ソース素材 | パターン抽出の一次資料 |

## ナレッジ追加のワークフロー

1. `packages/knowledge/` に事例ファイルを追加する
2. 「スケーラブルオーバーサイトへの示唆」セクションでパターンとの接続を記述する
3. 既存パターンに合致する → パターンファイルの「既知の使用例」に追記する
4. 新しいパターンを発見した → `app/patterns/` に新規ファイルを作成する

## パターンファイルの形式

各パターンは WikiXP（Portland Pattern Repository）形式に準拠する：

- **タイトル** — パターン名（h1）
- **サマリー** — 1〜2 文でパターンを要約（h1 直後の地の文）
- **コンテキスト** — このパターンが適用される状況
- **問題** — 解くべき問題を **太字** で記述
- **フォース** — 問題を難しくする競合する制約・緊張関係（箇条書き）
- **解決策** — パターンの解決策を **太字** で記述し、図や表を続ける
- **結果コンテキスト** — 適用後の状況（得られるもの・失うもの）
- **既知の使用例** — 実世界での適用事例
- **関連パターン** — 他パターンへのリンク
- **ソース** — 対応する `knowledge/` ファイルへのリンク

> **フォース** セクションが現在の形式との最大の差異。「なぜその解決策が難しいか」を規定する競合する制約を明示することで、パターンの適用判断を支援する。

## 分野について

分野（広告、コード、医療など）による分類は行わない。**パターン軸で整理する**ことを優先する。同じパターンが異なる分野に現れることがこのリポジトリの主要な発見になる。
